## 0. What this is

Two inseparable deliverables:

1. **The runtime** — a server-side TypeScript framework for turn-based, room-based multiplayer games where every genre-generic problem (rooms, reconnection, timers, state, recovery, observability) is solved once, to the standard of commitments C1–C12, permanently.
2. **The operating model** — the paved road: the workflow by which the author today, and a team tomorrow, builds, ships, debugs, and evolves games. The framework encodes the workflow; the tooling enforces it; the right way is the easy way.

A game built on this is a pure module: rules, invariants, projections. Everything impure — time, randomness, network, persistence, money — belongs to the framework.

## 1. North star — three measured windows

- **TTD (time-to-detect):** wrong-state occurrence → system detection.
- **TTE (time-to-engage):** detection → a human engaged.
- **TTRC (time-to-root-cause):** human engagement → precise root cause.

Targets, stated separately from definitions: TTRC **under 1 minute**; TTD and TTE targets are set from the Phase 1 baseline. All three are design tests for every feature. Phase 1 produces two named artifacts: the **incident baseline** (historical *actual* numbers for the last ~20 real production incidents, measured from tickets and logs) and the **coverage projection** (separately-labeled counterfactual: "would this architecture have caught it"). The projection's misses define the v1 observability backlog.

## 2. The constitution — twelve commitments

**C1. Three record classes, one truth.** The **match stream** (per room session): state-affecting events, consumed by reducers, hash-chained, fsync-durable; it also carries explicitly-typed zero-state-delta chained records (break-glass grants, seal-release, incident-settlement) — reducers ignore them, but they consume an eventSeq and extend the chain, making audit tamper-evident. The **admission log** (co-located): every inbound intent — player action, timer fire, connection transition — with server-assigned admission timestamp (which becomes the recorded event timestamp the injected clock exposes, per C8), verdict (accepted → eventSeq | rejected → machine-readable reason code), pre-decision state hash, and arbitration losers; ordered by (matchId, **incarnation**, lastAcceptedEventSeq, admissionSubSeq); group-fsync with a declared loss window (default 100 ms) covering only rejection/telemetry records — accepted intents are in the match stream. Heartbeat evidence is retained (default 30 days) so disconnect adjudications survive crashes. The **operational stream**: telemetry, weakest durability, never a replay input. Post-terminal records (break-glass on archived sessions, repro-export creation, re-bless acceptances) chain in a per-session **audit appendix** — a separate append-only chain whose genesis is the emitted settlement digest, written by a framework-level audit writer distinct from the room writer. "Pressed X, nothing happened" is answered by scrubbing the admission log.

**C2. Replay is verified, not assumed.** Game state is canonical plain JSON: objects, arrays, strings, integers, booleans, null — Map/Set/undefined/-0/floats rejected at the contract boundary. Every accepted event carries a post-apply canonical state hash; the replay chain verifies the post-apply sequence and hard-fails at the first mismatching eventSeq — replay can never complete silently wrong. Every chain anchor (segment header and each re-anchor point, C4) pins an environment fingerprint: framework version, game build hash, Node major + V8, lockfile hash, schema versions, PRNG algorithm id. Replay consumes recorded outcomes and never re-runs the PRNG. Purity is **detected, not enforced**: lint bans (Date, Math.random, Intl, locale methods, crypto, IO in reducer modules), dev-mode frozen-state double-execution with hash compare, and the hash-verified replay gate as the enforcement of record.

**C3. Randomness resolves at admission.** The PRNG (PCG32, pinned in the fingerprint) lives in the framework's admission layer, never in reducers. Events carry resolved outcomes (the drawn tile, the shuffle as consumed) **plus the cumulative PRNG stream offset** in the framework envelope — so crash recovery re-derives the segment seed, fast-forwards to the last recorded offset, and resumes; and fairness verification checks outcomes at recorded offsets without needing the game's draw-consumption mapping. Fairness is per segment: sealed masterSeed per session; segmentSeed = KDF(masterSeed, segmentIndex); segment-open records hash(segmentSeed), segment-close reveals it — every closed hand independently verifiable without exposing future hands. A **seal-release event** (revealing masterSeed for fairness verification of the unclosed segment) fires **only on session-terminating outcomes** — quarantine settlement, forced abort, abandoned-session expiry — never on a crash that recovers; no segment may open after seal-release. Repro never needs seeds: outcomes are in the events.

**C4. Logs outlive code: segments, version-runs, re-bless.** A session log divides into **segments** (hands/rounds; boundaries are ordinary events). Every event records its logicVersion; divergence classification operates per **version-run** (maximal run of same-version events): same-version mismatch = "data divergence (bug)"; cross-version = "code-changed (expected)" — never blended. A **verified snapshot at every segment boundary is constitutional**; cadence config adds mid-segment snapshots only. The hash chain re-anchors at each verified snapshot, and snapshot-vs-replay verification must pass before a new logicVersion appends. Deploys: snapshot at a segment boundary is the standard path; **mid-segment re-anchoring is a first-class path** for long-lived segments (correspondence, byo-yomi) — segment headers pin the ordered version list with re-anchor points. The deploy orchestrator must confirm the verified snapshot is durable **before** binary swap; crash-before-confirm rolls forward on the old binary. A closed-but-unsnapshotted segment is recoverable by same-version replay from the prior anchor — never voidable as "open". Validation is ingestion-only: replay never re-runs the validation pipeline. **Re-bless requires an executable change manifest** (machine-checkable declaration of which fields/invariants are expected to differ); out-of-manifest diffs block bulk-accept; at least one new fixture must positively demonstrate the intended change; manifest + acceptance are chained audit records. Re-bless materializes a **derived expectation set** (per-event hashes + terminal state under the current logicVersion) pinned to the original log digest; multi-version re-bless composes the manifest chain. **Original log bytes are immutable forever**: breaking schema changes produce derived views, never rewrites; tamper digests verify against originals; the migration CI predicate is replay-without-error + invariants hold + terminal equivalence under a semantics-preserving adapter (hash-identity is explicitly not the cross-migration predicate). **A single release changes reducer semantics or schema — never both**; manifests declare the schemaVersion their field paths refer to.

**C5. Durability & ownership.** An event commits only after fsync; no client ack and no external emission before the causing event commits. Records are length-prefixed and checksummed; recovery distinguishes clean from torn tails. **Quarantine is segment-granular**: settle at the last closed segment boundary (whose verified snapshot always exists, per C4); the open segment is voided. **Voided eventSeqs are never reused** — voided bytes are retained as a labeled dead branch and the void bumps the per-match **incarnation** counter; the incident-settlement event chains from the last verified snapshot with an explicit reference to the voided range. After quarantine resolution the session is closed — it never resumes play. A persisted tail behind last-acked seq is an incident, never a silent resume. One writer per room log, epoch/fencing tokens; v1 is single-node-per-room, multi-node = handoff, never concurrency. Shadow replay runs in a worker, never on the serving event loop.

**C6. Effects: replay can never re-fire money.** Execution mode — **live | recovery | shadow | local-replay | settling** — is an explicit constructor input, never env-derived. Non-live modes get a no-op effect recorder, structurally incapable of emission; **settling** (used for quarantine resolution and settlement corrections) has a real recorder but admission closed to player and timer intents, grants never armed, only framework incident/audit event types appendable, and transitions to closed — never to live. Recovery-mode replay never emits; on transition to live, the **outbox** — not the reducers — drains persisted intents above the durable high-watermark, **filtering out intents whose causing eventSeq lies in a voided range** (the void marker is durable before any transition). Idempotency key: **(matchId, incarnation, eventSeq, effectIndex)** with deterministic effect ordering; consumers dedupe on it. The settlement emission includes the chain digest as of settlement; post-settlement audit records are anchored by the audit appendix whose genesis is that digest.

**C7. Grants, not turns — one flow API.** The game supplies pure pendingGrants(state) → set of {seat, allowedActions, deadline, onExpiry, resolution}, evaluated after every event. **This is the single deadline API** — deadline derives from state; the framework arms it and appends exactly one expiry event per deadline; onExpiry names the default action, recorded as an ordinary admitted event. resolution defines arbitration for overlapping grants (priority | first-come | collect-all); losers land in the admission log. Replay consumes recorded expiry events at their log positions, never recomputing timers; restart re-derives pending deadlines from recovered state and re-arms. Rotation, claim windows, betting rounds are opt-in library helpers — never framework flow modes. Event vocabulary and all rendering speak grants; "current turn" is the degenerate single-grant view. The commit-reveal helper is built when its first consumer commits (pre-freeze paper designs count as consumers — the carve-out). **Pre-freeze gate:** poker (raise reopening, all-in, heads-up reversal), one sealed-bid game (exercising reveal-as-explicit-event), and one draft are paper-designed against the API; any inexpressible flow blocks the freeze.

**C8. Time & sessions.** The sole time authority visible to game logic is the server-assigned timestamp of the current recorded event; the injected clock returns exactly that. Reducers may hold time-derived values as ordinary state (chess clocks, time banks, byo-yomi); client time is never an input. A match log spans the room session; termination is defined per game (explicit close event or max-duration policy); **segments are the universal unit** for CI replay, seed commit/reveal, quarantine, and snapshots. External inputs (wallet responses) enter as recorded events. **Pre-freeze gate:** paper-replay a full hanchan with honba/riichi-stick/dealer-repeat carryover; a poker cash session with mid-session rebuy; a byo-yomi game with flag-fall and a restart over a live deadline.

**C9. One projection path, verified.** The game contract requires pure projectEvent(event, viewer) → redactedEvent | null and projectState(state, seat). Live broadcast AND reconnection resync derive from this one function; transport performs zero game-specific redaction. The commutation property — replaying the projected stream yields the same client view as projecting the replayed canonical state — is **verified** by a CI projection-replay gate, not assumed. Time-varying visibility uses **reveal-as-explicit-event**: the reveal event carries the newly-visible payloads; projection stays stateless; past events are never re-projected. **The framework owns projection of its own record types** (seal-release, break-glass, incident-settlement): default null for every viewer below break-glass (incident-settlement may define a player-facing notice); game projectEvent receives only game vocabulary. Observability gains per-seat replay and projection invariants ("seat A was never sent tile X before its reveal").

**C10. Information access: two artifact classes.** **Class A — sealed machine artifacts** (snapshots, crash payloads, recovery state, repro exports): full-fidelity by necessity, pseudonymized, encrypted at rest, access-audited, never human-rendered except break-glass. A **repro export** is identity-pseudonymized, payload-complete, chain-intact through the last verified anchor **plus an explicitly-labeled non-verifying tail** (voided events, torn-record bytes, seal-release) that replay tooling may consume for replay-to-first-divergence but that is excluded from chain verification; its creation is a chained audit event; it is the only valid replay input and the only artifact that becomes a fixture. **Seal-release payloads are stripped into an access-audited sealed sidecar at export time** (hash placeholder keeps the chain verifiable); fairness tooling is the sidecar's only consumer. **Class B — rendered surfaces** (replay views, inspector, alerts, viewing exports): always through the game's visibility rules for an explicit viewer role; never a replay input; alerts carry pointers, never payloads; no Class B surface shows seed, wall, or concealed hands of a live match outside break-glass. **Break-glass** is pre-authorized and instant for on-call — audit is a deterrent, not a latency gate — recorded as a chained event, reviewed post-hoc. **Identity:** opaque refs are the only player identifier in any framework-emitted stream, tag, dump, or alert; resolution happens solely at query time via the external crypto-shreddable map (erasure = shred the map, keep the log); the wallet's ledger stays outside the boundary. Reducers read nothing outside recorded state/log (lint/type-enforced), so pseudonymized exports replay identically by construction. The killer workflow: prod bug → repro export (Class A, audited) → local deterministic repro → fix → the export joins the regression corpus.

**C11. Detection & headless observability v1.** v1 ships: structured logs auto-tagged (opaque ref / matchId / eventSeq / logicVersion); black-box crash dump (Class A payload, Class B default rendering, pointer alert); repro and viewing export paths; CLI replay harness with replay-to-first-divergence, inline rejected intents, and field-diff filtering; invariant runner; support search (identity → map → opaque ref → player+time match search → one-command replay). **v1 page triggers, enumerated:** invariant violation, settlement-recompute mismatch, quarantine incident, data-divergence alert, torn-tail detection, crash. Settlement-touching reducers require a **redundant-recompute invariant** — an independently written recomputation compared at every settlement — the catch-all for valid-but-wrong states. **Projection invariant (constitutional):** no framework-internal record type ever reaches a Class B surface. Web timeline UI and live inspector are v2, gated on one game in production.

**C12. Corpora & CI economics — bounded by construction.** **Golden corpus** (curated, flow-diverse, every incident-derived repro export): replays per-commit; the pass predicate is hash-identity against each fixture's **latest blessed expectation set** (C4). **Nightly sample:** stratified recent production sessions. **Full archive:** retained per policy (incident-linked and golden: indefinite; others: sampled with a stated window); replayed in full weekly and before any release or migration. Cross-Node-major CI activates at the first Node upgrade; until then the fingerprint pins one version. A new game bootstraps by recording real sessions on the local harness and blessing them as initial goldens. Phase 1 retrofit logs are diagnostic-grade and pass a one-time retro-scrub before corpus admission.

## 3. The operating model

**Building a game** — the day-one loop: scaffold against the game contract (state shape, actions, invariants, pendingGrants, projections) → develop on the local CLI harness against stub primitives, replaying as you go → record real play sessions and bless them as initial goldens → drive the Shipping gates green. No networking, persistence, or observability code anywhere.

**Shipping a game** — gates: determinism gate green on the game's goldens; projection-replay gate green; a deliberately seeded leak caught by a projection invariant; settlement recompute present; goldens blessed. Deploy: graceful shutdown → verified snapshot (boundary, or mid-segment re-anchor for long segments) → orchestrator confirms snapshot durability → binary swap. Rollback: snapshot-recover on the previous logicVersion — the same mechanism, reversed.

**Handling an incident** — mitigation first, forensics second. Mitigation by class: quarantined match → settling mode, void open segment, settle at last closed boundary via the audited incident-settlement event through the outbox, session closes; wrong settlement emitted → correction emission through the same outbox/idempotency contract; stuck room → the deploy-grade snapshot-recovery path. Players are made whole through the money boundary **before** root-causing begins. Forensics: detect (page or support search) → replay at the divergent eventSeq → root cause → fix → re-bless with change manifest → repro export joins the corpus. Every production bug permanently strengthens the system.

**Evolving the framework** — the consumer rule: no feature without a named consumer currently blocked on it (a strangler-ported live subsystem, a committed game with a ship date, or — for contract helpers only — a pre-freeze paper design). Every core change replays the golden corpus in CI. **Distribution:** the framework is a versioned private package; every game is its own repository consuming it as a read-only dependency — framework source never lives in a game's tree, so forks are structurally impossible. Each game registers its golden fixtures with framework CI; **no framework version is released unless every registered game's goldens replay green** — the cross-game regression gate, enforced at release time. Games upgrade by version bump on their own schedule. The live mahjong server consumes the framework the same way during the strangler period, vendoring its goldens like any other registered game. The **frozen core** (event-log format, event- and wire-schema versioning mechanisms, replay harness, invariant API) changes only via charter amendment — ADR + version bump + corpus migration under the C4 predicate. The charter itself amends the same way.

**Scaling the team** — this charter is the constitution. A new engineer onboards by replaying real matches and reading ADRs, and cannot ship a game that bypasses determinism, projection, or invariants, because the game contract is the only API the framework offers.

## 4. The road

- **Phase 0 (now):** this charter; the C7/C8 paper-design gates.
- **Phase 1 — retrofit inside MJ-socket-server (first 3 months from charter approval):** per-match JSONL admission log, seed the ~4 RNG sites via recorded outcomes, last-200-events crash dump, CLI replay harness atop the existing emulation tests; produce the incident baseline + coverage projection. The live game gets diagnostic value immediately; the north star becomes falsifiable.
- **Phase 2 — Milestone 0 (two-week spike):** vertical-slice reducers for two rulesets (mahjong slice + one flow-divergent game) against the game contract on stubs. The contract survives both or changes now.
- **Phase 3 — framework v1 extraction (time-boxed one quarter; tiered):** *v1-blocking:* durability + torn-tail handling, hash chain + verified replay, effects gating + outbox, admission log, grant engine, single projection path, C11 observability, versioned release pipeline with the registered-goldens gate (Amendment 1). *Deferred with named stopgaps:* crypto-shredding (access-restricted logs), break-glass tooling (manual audited access), per-segment commit-reveal (sealed seed sidecar), cross-Node-major CI (pinned version), prod projection shadow-replay (dev + CI gates only). First strangler consumer in production: the grant/deadline engine replacing turnTimeoutHandler + claimOrchestrator.
- **Phase 4 — game #2:** flow-structurally different from mahjong; ships on v1; the Section 5 threshold test runs here; unlocks v2 (web timeline, inspector).
- **Deferred until demanded:** language-neutral protocol, client SDK, realtime engine, multi-node.

## 5. Success criteria — falsifiable

1. Phase 1 artifacts exist; at the Phase 4 gate, median TTRC over the re-classified incident set is under 1 minute, and TTD and TTE meet the baseline-derived targets.
2. Game #2 is flow-divergent and written as rules + invariants + projections only.
3. No silent divergence: every replay matches its anchored hash chain or hard-fails at a precise eventSeq.
4. Zero double-fired external effects across restarts, recoveries, replays, and incident settlements.
5. A seeded information-leak bug is caught by a projection invariant in CI.
6. Golden-corpus CI green on every core change; every re-bless carries a change manifest; no bulk-accept outside manifest scope.

## 6. Out of scope (v1)

Client SDK; realtime/tick games; payments/wallet internals (the effects contract and idempotency keys define the boundary; the API server owns money and identity ledgers); advanced matchmaking; web UIs (v2); language-neutral protocol; multi-node concurrent ownership; replicated-ack durability.

## Appendix — what this charter deliberately does not define

These are sequenced downstream on purpose; defining them now would be speculation the gates below might invalidate.

1. **Phase 0 paper gates (next, due now):** poker / sealed-bid / draft designed against the grant API (C7); hanchan-carryover / poker-rebuy / byo-yomi-restart paper-replays (C8). The grant API is a hypothesis until these run.
2. **The tech spec (after the gates):** concrete TypeScript interfaces, event envelope schema, log file framing, snapshot format, manifest schema, canonical-hash and KDF algorithm choices.
3. **The Phase 1 retrofit plan (lives in MJ-socket-server):** concrete call sites, JSONL shape, incident-classification rubric.
4. **A name** for the framework.

## Appendix — flagged design choices

- **Mid-segment re-anchoring** was chosen over capping segment wall-time, to keep correspondence-style games in scope; the cost is per-version-run (rather than per-segment) divergence classification.
- **One release, one axis** (reducer semantics or schema, never both) constrains release operations; deliberate, to keep manifests and migration adapters honest.
